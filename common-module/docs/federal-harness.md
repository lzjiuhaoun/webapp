# Federal Harness Engineering — 联邦式文档原则结构

> 大型工程中"全知全能根节点"会导致信息过载和耦合。联邦式（Federal/Recursive）结构通过**信息隔离**让 Agent 只需阅读"根目录 + 所属模块目录"即可工作，无需理解其他模块的内部细节。

## 核心逻辑

**根目录负责"原则与规范"，模块目录负责"封装与实现"。**

---

## 问题—方案链（需求→功能→技术实现）

联邦文档结构的核心骨架是**逐层收敛的问题—方案链**：每一层回答一个"做什么"，下一层回答"怎么做"，后者是前者的方案、同时又是更下层的"做什么"。

```
需求层（问题：用户遇到的痛点/场景）
    ↓ 方案化
功能层（方案：产品提供哪些能力来解决需求）
    ↓ 技术化
技术架构层（方案：系统如何分解、模块如何协作）
    ↓ 实例化/分配
模块接口层（方案：每个模块对外承诺什么）
    ↓ 实现
模块内部层（方案：模块内部怎么做到的）
```

### 各层的文档映射

| 层级 | 核心问题 | 对应文档 |
|------|---------|---------|
| **需求层** | 用户有什么痛点？在什么场景下需要什么？ | `REQUIREMENTS.md` |
| **功能层** | 产品提供什么功能来解决这些需求？ | `FUNCTIONS.md` |
| **技术架构层** | 系统由哪些模块组成？之间如何通信？ | `SYSTEM_ARCHITECTURE.md`、`GLOBAL_CONTRACTS.md` |
| **模块接口层** | 每个模块对外提供什么接口？ | 各 `EXPORTS.md` |
| **模块内部层** | 模块内部怎么实现其接口？ | 各 `INTERNAL_LOGIC.md` |

### 分解关系

每一层对上一层的关系是**方案对问题**，对下一层的关系是**问题到方案**：

- `REQUIREMENTS.md` 定义**问题域**（痛点、场景）
- `FUNCTIONS.md` 是问题域的**产品方案**：通过哪些功能覆盖需求
- `SYSTEM_ARCHITECTURE.md` 是功能层的**技术方案**：架构如何支撑功能
- `EXPORTS.md` 是架构层的**接口方案**：模块间的边界与契约
- `INTERNAL_LOGIC.md` 是接口层的**实现方案**：模块内如何履约

这保证了**方案始终对齐问题**，避免过度设计或需求漂移。

## 一、根目录（Level 0: System Governance）

根目录文档定义系统的**整体边界**和**各主权模块之间的规范协议**。

### `CLAUDE.md` — 系统宪法
- **定义**：项目的最高入口文档。
- **职责**：定义项目的宏观使命、全局强制阅读顺序、联邦模块清单、全局冲突解决优先级。

### `SYSTEM_ARCHITECTURE.md` — 系统架构
- **定义**：从功能清单推导的系统架构图。
- **职责**：描述模块分解、分技术、分接口、存储拓扑、子模块角色矩阵。

### `GLOBAL_CONTRACTS.md` — 规范条约
- **定义**：跨模块通信的硬性保证。
- **职责**：定义全局唯一标识符规范、模块间边界承诺。规定"谁必须给谁提供什么"，而非"怎么提供"。

### `SHARED_DOMAIN.md` — 通用语言
- **定义**：全系统共享的核心实体定义。
- **职责**：定义核心领域对象的语义和生命周期状态，确保所有模块对同一概念有统一认知。

### `INTEGRATION_TASKS.md` — 集成里程碑
- **定义**：跨模块的协同进度表。
- **职责**：从各模块 `TASKS.md` 提取，仅关注集成测试点，不记录模块内部琐碎开发任务。

### `NON_FUNCTIONAL.md` — 全局硬约束
- **定义**：所有模块必须遵守的技术约束。
- **职责**：列出禁止项（无数据库、无消息队列、无微服务等），防止过度工程。

### `ACCEPTANCE.md` — 端到端验收
- **定义**：全系统集成后的验收场景。
- **职责**：端到端场景覆盖需求，通过即系统可交付。

## 二、模块目录（Level 1+: Module Autonomy）

每个模块拥有自己独立的指令集，像一个独立的微型工程。所有子模块按同一模式递归。

### `[MODULE]/CLAUDE.md` — 模块指令
- **定义**：该模块 Agent 的直接入口。
- **职责**：声明该模块采用的具体技术栈和本地的 Source of Truth。

### `[MODULE]/EXPORTS.md` — 出口契约
- **定义**：该模块对外承诺的接口文档（版本化）。
- **职责**：列出本模块提供的服务/接口/端点签名，对其他模块是唯一真相源。

### `[MODULE]/INTERNAL_LOGIC.md` — 内部实现逻辑
- **定义**：该模块的"黑盒"内幕。
- **职责**：详细描述模块内部的处理流程。这些信息对根目录和其他模块**不可见**。

### `[MODULE]/DEPENDENCIES.md` — 依赖声明
- **定义**：本模块的外部依赖和内部依赖。
- **职责**：声明消费了哪些其他模块的 `EXPORTS.md`（含版本约束），以及外部库/工具依赖。

### `[MODULE]/TESTING.md` — 测试策略
- **定义**：模块内的测试方案。
- **职责**：工具链、Mock 边界、测试目录结构。

### `[MODULE]/TASKS.md` — 局部任务书
- **定义**：模块内的开发序列。
- **职责**：承载具体的分阶段开发任务。

### `[MODULE]/ACCEPTANCE.md` — 模块验收
- **定义**：模块内的功能验收标准。
- **职责**：该模块维度的场景覆盖，通过后提交集成。

### `[MODULE]/xxx/` — 递归子模块
- 若模块内部可进一步分解为子域，则子域同样拥有 `CLAUDE.md`、`EXPORTS.md`、`INTERNAL_LOGIC.md`、`DEPENDENCIES.md`（若有）、`TESTING.md`（若有）、`ACCEPTANCE.md`（若有）。

## 三、协作机制

| 原则 | 说明 |
|------|------|
| **黑盒原则** | 模块 A 想增加功能时，只读 `GLOBAL_CONTRACTS.md` 确认接口位置 + 读本模块 `CLAUDE.md` 实现；**绝不**需要读其他模块的 `INTERNAL_LOGIC.md` |
| **契约驱动** | 任何影响他模块的变更，必须先更新 `GLOBAL_CONTRACTS.md` 或所属模块 `EXPORTS.md`（并提升版本号），再改代码 |
| **递归验证** | 每个模块有自己的 `ACCEPTANCE.md`；根 `ACCEPTANCE.md` 只负责最后的端到端集成验收 |
| **引用不复制** | 跨模块依赖用 `DEPENDENCIES.md` 相对路径引用，唯一真相源在提供方目录 |
| **文档图完整性** | 所有被引用的文件必须真实存在；从 `CLAUDE.md` 和 `AGENTS.md` 出发必须能追踪到所有文档。新增/删除文档时须更新入口引用 |

## 四、适用条件

这套结构适合：
- 多模块/多 Agent 协作的大型工程
- 需模块间强隔离、契约化通信的场景
- 工程复杂度超出单个 Agent 上下文窗口极限

不适合：
- 单模块小项目（过度工程）
- 团队紧密耦合、无需信息隔离的场景
